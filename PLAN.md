# PLAN.md
## Поэтапный план реализации приложения GoNext — Дневник туриста

---

## Этап 0. Подготовка и инфраструктура

1. Убедиться, что проект на Expo Router + TypeScript + React Native Paper собран и запускается (уже сделано).
2. Добавить зависимости для работы с данными:
   - SQLite (например, `expo-sqlite`)
   - при необходимости — пакеты для работы с файлами и медиа (фото).
3. Спроектировать и описать схему БД (таблицы, индексы) в отдельном документе или в коде миграций.

---

## Этап 1. База данных и модели

1. Создать и инициализировать локальную БД (expo-sqlite).
2. Реализовать миграции/схему:
   - таблица **places**: id, name, description, visitlater, liked, latitude, longitude (dd), createdAt;
   - таблица **place_photos**: id, placeId, path/uri, order;
   - таблица **place_contacts**: id, placeId, тип контакта, значение;
   - таблица **trips**: id, title, description, startDate, endDate, createdAt, current;
   - таблица **trip_places**: id, tripId, placeId, order, visited, visitDate, notes;
   - таблица **trip_place_photos**: id, tripPlaceId, path/uri, order.
3. Вынести работу с БД в слой доступа (сервисы/репозитории) для places, trips, trip_places.
4. Определить TypeScript-типы/интерфейсы для Place, Trip, TripPlace и связанных сущностей.

---

## Этап 2. Навигация и главный экран

1. Настроить структуру маршрутов Expo Router:
   - корневой layout (например, Stack или общий layout с табами/дроваером — по выбору);
   - маршруты: главный экран, места, поездки, следующее место, настройки.
2. Реализовать **главный экран** (Home):
   - заголовок «GoNext» (AppBar);
   - четыре кнопки: «Места», «Поездки», «Следующее место», «Настройки»;
   - переход по кнопкам на соответствующие экраны.
3. Создать заглушки экранов: Список мест, Список поездок, Следующее место, Настройки (минимальный UI без логики).

---

## Этап 3. Режим «Места»

1. **Список мест**:
   - экран со списком (FlatList/SectionList или аналог из Paper);
   - загрузка данных из БД через сервис мест;
   - кнопка «Добавить место» (переход на форму создания/редактирования).
2. **Создание и редактирование места**:
   - форма: название, описание, visitlater, liked, координаты (dd);
   - сохранение в БД (insert/update);
   - опционально: выбор точки на карте для заполнения координат.
3. **Детальная карточка места**:
   - просмотр всех полей места;
   - список фотографий; добавление/удаление фото (сохранение файлов локально, запись путей в БД);
   - список контактов; добавление/редактирование/удаление контактов;
   - кнопка «Открыть на карте» (переход в приложение карт по координатам).
4. При необходимости вынести общую логику «открыть на карте» в утилиту (Linking + geo-URL).

---

## Этап 4. Режим «Поездки»

1. **Список поездок**:
   - экран со списком поездок (из БД);
   - отображение title, дат, признака current;
   - кнопка «Создать поездку».
2. **Создание и редактирование поездки**:
   - форма: title, description, startDate, endDate, current;
   - сохранение в БД.
3. **Детали поездки (маршрут)**:
   - отображение упорядоченного списка мест поездки (TripPlace);
   - добавление мест в поездку: выбор из существующих Place или создание нового;
   - изменение порядка мест (drag-and-drop или кнопки вверх/вниз);
   - для каждого места в поездке: отметка «посещено» (visited), visitDate, notes, photos;
   - сохранение изменений в таблицах trip_places и trip_place_photos.
4. Реализовать логику «текущей поездки» (current): при создании/выборе текущей — сбрасывать current у остальных поездок.

---

## Этап 5. Режим «Следующее место»

1. Реализовать сервис/функцию получения «следующего места»:
   - найти поездку с current = true;
   - среди её trip_places выбрать первое с visited = false (по order);
   - вернуть соответствующее Place и контекст (поездка, порядок).
2. **Экран «Следующее место»**:
   - если есть текущая поездка и следующее место — показать карточку места (название, описание, координаты);
   - кнопки: «Открыть на карте», «Открыть в навигаторе» (Google Maps / системный навигатор по координатам);
   - если следующего места нет — показать сообщение (нет текущей поездки или все места посещены).
3. Использовать Linking для открытия карт и навигации (geo URI, Google Maps URL и т.п.).

---

## Этап 6. Настройки и полировка

1. **Экран «Настройки»**:
   - минимальный набор опций (например, о приложении, версия);
   - при необходимости — настройки единиц, языка и т.д. (по требованиям MVP).
2. Единообразие UI: все экраны на компонентах React Native Paper, общий стиль навигации (кнопки «Назад», заголовки).
3. Обработка краевых случаев: пустые списки мест/поездок, отсутствие текущей поездки, отсутствие следующего места.
4. Проверка офлайн-работы: отключить сеть и убедиться, что основные сценарии работают без интернета.

---

## Этап 7. Тестирование и подготовка к релизу

1. Ручное тестирование основных сценариев на Android и iOS (или в Expo Go).
2. Проверка хранения фото и размера данных при большом количестве мест и поездок.
3. При необходимости — создание development build (EAS Build или локально) для проверки с нативными модулями (SQLite, файловая система).
4. Подготовка краткой инструкции по установке и использованию (для себя или для тестировщиков).

---

## Порядок этапов (кратко)

| № | Этап |
|---|------|
| 0 | Подготовка: зависимости, схема БД |
| 1 | БД и модели (таблицы, типы, сервисы) |
| 2 | Навигация и главный экран с кнопками |
| 3 | Режим «Места» (список, карточка, фото, контакты, карта) |
| 4 | Режим «Поездки» (список, маршрут, порядок, посещения, заметки, фото) |
| 5 | Режим «Следующее место» (логика + экран + карта/навигатор) |
| 6 | Настройки и полировка UI |
| 7 | Тестирование и подготовка к релизу |

Реализацию удобно вести строго по этапам: сначала стабильная БД и навигация, затем режимы «Места» → «Поездки» → «Следующее место», в конце настройки и тесты.
